4) الخادم (Backend)

برنامج Node.js/Express مبني بطبقات منفصلة:

  • routes/ — نقاط النهاية (الدخول، الطلبات، التصاريح، المستخدمون، إدارة واتساب).
  • middleware/ — بوابات قبل المعالجة (تحقّق الجلسة، الصلاحيات، رفع الملفات، الأخطاء).
  • services/ — منطق العمل (حالات الطلب، الإصدار، التدقيق، الإشعارات).
  • db/ — الوصول لقاعدة البيانات والمخطّط.

5) الواجهة (Frontend)

تطبيق صفحة واحدة (SPA) بـJavaScript نقي بلا أطر ثقيلة: ملف واحد index.html + app.js يرسم الشاشات ويتواصل مع الخادم عبر /api، وstyles.css للتصميم. لا إعادة تحميل — يتغيّر المحتوى فقط.

6) قاعدة البيانات

SQLite في ملف واحد (pams.db) عبر مكتبة better-sqlite3، بوضع WAL للأمان، على قرص دائم منفصل عن الكود (/var/pams-data/data/).

مسار الطلب عبر طبقات الحماية:

User → Nginx → Express → Helmet/CSP → Rate-Limit → Auth (JWT + Session) → RBAC → Route → Database → Audit
الخطوة القادمة: تركيب HTTPS (شهادة SSL مجانية) لتشفير الاتصال — جاهز للتركيب فور تجهيز نطاق (دومين).

8) كيف يعمل الوكيل (Agent)

الفكرة الأساسية: استخدام طابور مهام داخل قاعدة البيانات لضمان الموثوقية (لا يفقد عملاً ولا يتعطّل عند الفشل).

① استقبال ملف عبر whatsapp-web.js ② تحقّق القائمة المسموحة ③ حفظ سجلّ + وضع مهمة في الطابور (wa_jobs) ④ عامل (Worker) ينفّذ المهمة — مع إعادة محاولة وعزل للفشل ⑤ قراءة الهوية والتاريخ (OCR.space / tesseract) ⑥ مطابقة بطلب موجود → إصدار التصريح ⑦ توزيعه للمهندس على واتساب + رابط دخول ⑧ تقرير موحّد (صُدِر / سبق إصداره / غير موجود)
  • الاتصال: whatsapp-web.js يشغّل متصفّح Chromium مخفي مسجّل على رقم واتساب (الجلسة محفوظة).
  • الموثوقية: طابور في قاعدة البيانات → يبقى بعد إعادة التشغيل + إعادة محاولة + عزل الفشل (Dead Letter).
  • الفصل: الاستقبال ≠ المعالجة ≠ التوزيع، وكل جزء له تحديد معدّل مستقل (تفادي حظر واتساب).

9) توصيات التحسين

  • تفعيل HTTPS (الأولوية القصوى) عبر نطاق + شهادة SSL مجانية.
  • إضافة Cloudflare (مجاني): إخفاء عنوان الخادم + حماية DDoS + جدار حماية.
  • نسخ احتياطي مجدول تلقائي لقاعدة البيانات.
  • تقييد الوصول للخادم وقاعدة البيانات بقوة.
